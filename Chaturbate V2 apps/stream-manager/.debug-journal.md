## Bugs Found & Fixed

### 1. DUPLICATE `const` Declarations (CRITICAL — SyntaxError in production)
- **File**: `shared.js`
- **Issue**: `ANON_BLACKLIST_KEY`, `PANEL_CYCLE_LABEL`, and `PANEL_VIEW_KEY` were declared twice with `const`. In strict mode / ES6, redeclaring a `const` throws a `SyntaxError`, which would crash the entire app in the Chaturbate IDE sandbox.
- **Fix**: Removed the second block of duplicate declarations. Kept the first occurrence (near related constants).

### 2. Non-Atomic Counter Increment (`incrCounter`)
- **File**: `shared.js`, `simulator.js`
- **Issue**: `incrCounter()` used a read-modify-write pattern (`get` → add → `set`), which can lose increments under race conditions if multiple events fire simultaneously (e.g., a tip storm with rapid-fire tips).
- **Fix**: Replaced with `$kv.incr()` — the platform's atomic increment method. The function now returns `$kv.get(key, 0)` to retrieve the new value after atomic increment.

### 3. Unbounded KV Growth (Scroll Offset & Animate Tick)
- **File**: `shared.js`, `simulator.js`
- **Issue**: `getNextScrollOffset()` and `getNextAnimateTick()` incremented KV values without any upper bound. Over hours-long broadcasts, these keys grow unbounded, wasting storage and slowing iteration.
- **Fix**: Wrapped at 1000 (scroll) and 100 (animate) using modulo. Values cycle, preventing unbounded growth while maintaining functional correctness.

### 4. Duplicate Public Room Notices
- **Files**: `handlers/tip-received.js`, `handlers/fanclub-join.js`, `simulator.js`
- **Issue**: Multiple handlers called `$room.sendNotice()` directly AND then called `addNotification()`, which ALSO calls `$room.sendNotice()` internally. This caused each event to produce two identical public notices in chat — once from the handler, once from the rolling notification system.
- **Fix**: Removed the direct `$room.sendNotice()` calls in `tip-received.js` and `fanclub-join.js`. The `addNotification()` function is now the single source of public announcements for these events. Tip message content is preserved in the notification text.

### 5. Hardcoded Personal Affiliate Links in Default Templates
- **Files**: `shared.js`, `simulator.js`, `settings-schema.json`
- **Issue**: The `REPLY_TEMPLATES` and default notification reply template settings contained hardcoded Fansly URLs specific to one creator (`https://fansly.com/Peralez_f/t0`). These are personal marketing links that don't belong in shared default templates.
- **Fix**: Replaced with generic, broadcaster-agnostic reply templates that redirect to `!menu` command. Broadcasters can customize via settings if they want affiliate links.

### 6. Missing `hiddenCamPrice` Setting in Schema
- **File**: `settings-schema.json`
- **Issue**: The `app-settings-change.js` handler references `$settings.hiddenCamPrice` and `$settings.hiddenCamMessage`, but these settings were never defined in the IDE schema. This means broadcasters couldn't configure them, and the settings would always return `undefined`.
- **Fix**: Added both `hiddenCamPrice` (number, default 100) and `hiddenCamMessage` (text, default with `{{price}}` placeholder) to the settings schema.

### 7. Unused Variables
- **File**: `simulator.js`, `handlers/broadcast-panel-update.js`
- **Issue**: `loadHandlersCalled` in simulator.js was initialized and set but never read. `blCount` and `anonCount` in broadcast-panel-update.js were declared but never displayed.
- **Fix**: Removed `loadHandlersCalled`. Repurposed `blCount` and `anonCount` into the Anon Conv panel view (View 1) to show real-time anon/blacklist counts.

## Round 2 Fixes (v1.1.0 — 2026-07-16)

### 8. `!anon duration` Not Persisted to KV
- **File**: `handlers/chat-message.js`
- **Issue**: The `!anon duration <sec>` command set `$settings.anonConverterBlackoutDuration` in-memory but never persisted the value to KV. Compare with `!anon interval` which correctly used `$kv.set(ANON_CONVERTER_INTERVAL_KEY, sec)`. Duration changes were lost on app reload.
- **Fix**: Added `$kv.set(ANON_CONVERTER_DURATION_KEY, dur)` after the in-memory assignment.

### 9. `app-stop.js` Missing `stopNotificationAutoTimer()`
- **File**: `handlers/app-stop.js`
- **Issue**: All other timers (`tipGoalReminder`, anonConverter, panelCycle) were explicitly stopped in `onAppStop`, but the notification auto timer was not. This left the timer running after app stop.
- **Fix**: Added `stopNotificationAutoTimer()` call after `stopPanelCycle()`.

### 10. Panel Reload Debouncing
- **Files**: `shared.js`, `handlers/callback.js`, `handlers/tip-received.js`, `simulator.js`
- **Issue**: Every tip triggered `$room.reloadPanel()` directly. In a tip storm, this caused rapid panel re-renders and visual flickering. The debug journal noted this as a deferred optimization.
- **Fix**: Added a `debouncedReloadPanel()` function that cancels any pending reload and schedules a new one via a 1-second one-shot callback. The callback handler was added to `onCallback`. Tip events now use the debounced variant.

### 11. Anon Tip Goal Visibility
- **Files**: `handlers/chat-message.js`, `simulator.js`
- **Issue**: Anonymous tips were tracked in the `anonTipTokens` KV key but never surfaced in any user-facing output. The `!goal` command and `!anon` status display didn't show anon contribution.
- **Fix**: `!goal` now appends `| Anon: <amount>` when anon tips exist. `!anon` status now includes `Anon tips: <amount>` alongside the existing fields.

### 12. Simulator Codebase Drift (Mismatches from shared.js)
- **File**: `simulator.js`
- **Issues**:
  - `incrCounter` still used read-modify-write pattern (pre-fix #2 from Round 1)
  - `getNextScrollOffset` / `getNextAnimateTick` didn't wrap at modulo bounds (pre-fix #3)
  - `resetNotificationSettings` didn't reset notification or reply templates (missing template resets from shared.js)
- **Fix**: All three functions updated to match shared.js exactly.

### 13. Deterministic Simulator
- **File**: `simulator.js`
- **Issue**: The simulator used `Date.now()` and `Math.random()` directly, producing different output on every run. This made regression testing by diff impossible.
- **Fix**: Overrode `Date.now` with a fixed-base timestamp that increments by 1 second per call. Overrode `Math.random` with a Park-Miller LCG seeded at 42. Every run now produces identical output.

## Optimizations Applied

1. **Atomic KV Operations**: All counter increments now use `$kv.incr()` — safer under concurrent event delivery.
2. **Bounded KV Iteration**: Scroll offset and animate tick now wrap at sane upper limits instead of growing indefinitely.
3. **Single Source of Truth for Notices**: Removed duplicate `$room.sendNotice()` calls. The `addNotification()` system is now the centralized path for public event announcements.
4. **Cleaner Default Templates**: Removed personalized content from shared defaults. Templates are now broadcaster-agnostic.
5. **Panel Reload Debouncing**: Rapid-fire tip events no longer cause panel flickering. Reloads are coalesced into one call per second.
6. **Deterministic Simulator**: Enables diff-based regression testing between versions.

## Round 3 Fixes (v1.2.0 — Deep Debug & Cross-Feature Conflicts)

### 14. `formatTipMenuText()` Padding Broken (Cosmetic)
- **Files**: `shared.js`
- **Issue**: The alignment padding loop pushed space characters as separate array elements (`parts.push(' ')`), then only used `parts[0]` for the final line. The padding spaces were discarded, producing unaligned output.
- **Fix**: Changed to string concatenation: build `paddedName` directly with inline spaces, construct the full `line` string once, and push to `lines`.

### 15. Hidden Cam / Anon Converter Mutual Exclusion
- **Files**: `shared.js`, `simulator.js`, `handlers/chat-message.js`
- **Issue**: Both hidden cam and anon converter share a single `$limitcam` instance. The anon converter correctly checked `$kv.get(HIDDEN_CAM_KEY, false)` before starting a blackout, but the reverse was not protected:
  - `startHiddenCam()` didn't check if an anon converter blackout was active — could overlap.
  - `doAnonConverterRestore()` called `$limitcam.stop()` unconditionally — if hidden cam was started during a blackout, restore would accidentally kill it.
- **Fix**:
  - `startHiddenCam()` now checks `$kv.get(ANON_CONVERTER_BLACKOUT_KEY, false)` and refuses to start with a broadcaster notice.
  - `doAnonConverterRestore()` only calls `$limitcam.stop()` when hidden cam is NOT active (`!$kv.get(HIDDEN_CAM_KEY, false)`).

### 16. `$limitcam.users` Unsafe Access (Potential Production Crash)
- **Files**: `handlers/broadcast-panel-update.js`, `handlers/chat-message.js`, `handlers/shortcut.js`, `simulator.js`
- **Issue**: Multiple handlers referenced `$limitcam.users.length` without guarding for the case where `$limitcam.users` might be `undefined`. In the Chaturbate V2 sandbox, `$limitcam.users` might not be exposed as a public property, causing a `TypeError` at runtime.
- **Fix**: Changed all occurrences to `($limitcam.users || []).length`.

## Items Left for Future Iteration (updated)

1. **Simulator Import Architecture**: simulator.js currently duplicates ~95% of shared logic inline. Any change to shared.js or handlers requires mirroring in simulator.js. Consider refactoring to `require()` shared modules when running in Node.js.

2. **`$message.orig` Preservation**: The `onChatMessageTransform` handler should preserve `$message.orig` (the original body before transformation) for downstream debugging, as documented in the API. Currently the handler only sets spam without modifying the body, so `$message.orig` is preserved by the platform automatically.

## Round 4 Fixes (v1.2.0 — Remaining Journal Items)

### 17. Goal Reset Race Condition (Item 3)
- **Files**: `shared.js` (GOAL_REACHED_KEY constant), `handlers/tip-received.js`, `handlers/broadcast-start.js`, `simulator.js`
- **Issue**: When a tip pushes past the goal, the total resets to 0. If two tips arrive simultaneously and both cross the threshold, the goal-reached notice fires twice.
- **Fix**: Added `GOAL_REACHED_KEY` guard. Before resetting, tip-received checks `$kv.get(GOAL_REACHED_KEY, false)` and returns early if already set. After reset, sets the flag to `true`. The flag is cleared on `onBroadcastStart` (new session). Simulator mirrors the change.

### 18. Overlay Auto-Reconnect (Item 4)
- **Files**: `shared.js` (emitFullStateSync), `handlers/app-start.js`, `overlay.html` (onStateSync), `simulator.js`
- **Issue**: The OBS overlay loses state when OBS scene toggles or the overlay reconnects — no mechanism to re-establish current progress, top tipper, hidden cam/anon status.
- **Fix**: Added `emitFullStateSync()` in shared.js that `$overlay.emit('stateSync', {...})` with full current state (tipGoal, currentTips, percentage, topTipper, sessionStart, hiddenCamActive, anonConverterActive, followerCount). Called from `onAppStart`. In overlay.html, added `onStateSync` handler that calls `updateBar(data.currentTips, data.tipGoal)` to re-establish the progress bar on reconnect. Simulator mirrors the change.
